# I Built a Complete Enterprise GCP Infrastructure with Terraform — Here's Everything I Learned

## The Goal

Deploy production-grade, multi-application infrastructure on Google Cloud Platform using Terraform — the kind of setup you'd actually find at a real company, not a tutorial with a single VM.

The result: 64 cloud resources across 6 modules, verified with an automated script, deployable in under 20 minutes from scratch.

## What I Built

Three applications sharing common infrastructure:

**Web Application** — A 3-tier web app with a Global HTTPS Load Balancer fronting a regional Managed Instance Group of nginx servers that autoscale based on CPU utilization. Instances have no external IPs — they receive traffic only through the LB.

**Data Pipeline** — An event-driven ETL pipeline: files uploaded to a GCS landing bucket trigger a Pub/Sub notification, which invokes a Cloud Function v2 (Python) that parses the file, writes records to a partitioned BigQuery table, and moves the file to a curated bucket.

**API Microservice** — A Cloud Run v2 service with Artifact Registry for container storage. Serverless, scales to zero, auto-scales under load.

All three share: a custom VPC with public/private subnets, Cloud NAT for outbound internet, IAP-only SSH access, dedicated least-privilege service accounts, and a Cloud SQL PostgreSQL database with private IP only.

## The Architecture Decisions That Matter

### Remote State from Day One

Every serious Terraform project uses remote state. I used a GCS bucket with versioning enabled. This means:
- State is never lost if your machine dies
- Multiple engineers can work on the same infrastructure
- State locking prevents concurrent applies from corrupting state
- You can roll back to any previous state version

### Module Composition via Output Chaining

The real power of Terraform modules isn't code reuse — it's explicit dependency management through output chaining:
```hcl
module "compute" {
  vpc_name              = module.networking.vpc_name
  service_account_email = module.iam.web_app_sa_email
}
```

Terraform reads these references and builds a dependency graph automatically. It knows networking must complete before compute, IAM before compute, all without you specifying any explicit ordering.

### Least Privilege IAM — For Real

Most tutorials grant `roles/editor` to a service account and move on. I created a dedicated service account for each workload with only the roles it needs:

- Web app: `logging.logWriter` + `monitoring.metricWriter`
- Data pipeline: `bigquery.dataEditor` + `storage.objectAdmin` + `pubsub.editor`
- API service: `secretmanager.secretAccessor` + `logging.logWriter`

If any workload is compromised, the blast radius is limited to exactly what that service account can do.

### Private Database — No Exceptions

The Cloud SQL instance has `ipv4_enabled = false`. No public IP. Period. It's only reachable from within the VPC via private IP (`10.51.0.3`), enabled by `google_service_networking_connection` which peers your VPC with Google's managed service network.

The password is generated by `random_password`, immediately stored in Secret Manager, and never appears in logs or plan output.

### Content-Addressable Function Deployment
```hcl
resource "google_storage_bucket_object" "function_zip" {
  name = "etl-function-${data.archive_file.function_zip.output_md5}.zip"
}
```

The zip filename includes its MD5 hash. Every time the Python code changes, the hash changes, the object name changes, and Terraform uploads a new version and updates the function. No manual deployment steps, no version drift.

## What Surprised Me

**Cloud Shell network instability** — The GCP metadata server for authentication occasionally fails with `EOF` errors. The fix: always use `gcloud auth application-default login` and set `GOOGLE_APPLICATION_CREDENTIALS` explicitly. Add it to `.bashrc`.

**Cloud NAT scope matters** — I initially configured NAT for the private subnet only. Instances in the public subnet (which also had no external IPs) couldn't reach the internet to install nginx. Changed to `ALL_SUBNETWORKS_ALL_IP_RANGES`.

**Cloud Functions v2 needs Eventarc API** — Terraform gives you a clear 403 error. Just enable it: `gcloud services enable eventarc.googleapis.com`.

**instance template names need prefixes** — Instance templates are immutable. Use `name_prefix` instead of `name` so Terraform can create a new one before destroying the old one (`create_before_destroy = true`).

## Lessons for Your Own Projects

1. **Always use remote state** — local state is fine for learning, never for real work
2. **One service account per workload** — never share, never use default compute SA
3. **Private IPs for databases** — no exceptions, even in dev
4. **Validate your variables** — catch `environment = "production"` before it reaches GCP
5. **Use `lifecycle { create_before_destroy = true }`** on instance templates — zero downtime updates
6. **Tag everything** — `labels = { environment = var.environment, managed_by = "terraform" }`
